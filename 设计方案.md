## 1. 项目背景与目标

### 1.1 原始项目概述

原始仓库（Node.js 版本）位于 `./evolver/`，是 EvoMap 的 GEP (Genome Evolution Protocol) 自进化引擎，版本 **v1.89.14**。它是一个生产级、高度工程化的系统：

- **172 个 JavaScript 源文件**，分布于 `src/` 下 7 大子系统
- **164 个测试文件**，使用 Node.js 内置 `node --test`，覆盖单元、集成、安全、韧性、并发场景
- **18 个运维脚本**，涵盖构建分发、A2A 资产治理、版本管理、崩溃恢复
- **多语言 README**（英/中/日/韩）+ 详细的 SKILL.md（365 行，描述 Proxy Mailbox API）
- **核心算法受保护**：52 个文件经 `javascript-obfuscator` 混淆，包括进化管道与 GEP 核心

当前项目 `evolver.py` 是其 Python 3.12+ 移植版本，目标是**完全行为等价**。它通过扫描 `memory/` 目录中的运行时日志、错误模式、用户反馈，从本地 GEP 资产库中选择匹配的 Gene 或 Capsule，生成协议约束的 GEP prompt，并记录可审计的 EvolutionEvent。

### 1.3 移植目标

创建 `./evolver.py/`，实现：

1. **CLI 行为等价**: 相同的命令、参数、环境变量、退出码。
2. **数据格式等价**: 完全兼容 `genes.json`/`genes.jsonl`、`capsules.json`/`capsules.jsonl`、`events.jsonl`、`memory_graph.jsonl`、`evolution_solidify_state.json` 等。
3. **文件路径等价**: 相同的默认路径解析（`~/.evomap/`、`~/.evolver/`、`.evolver/`、`memory/`、`assets/gep/`）。
4. **核心流水线等价**: signals → collect → hub → select → enrich → dispatch 的上下文传递和副作用。
5. **本地 Proxy 等价**: 路由、mailbox、sync engine、heartbeat、模型路由。
6. **ATP 等价**: 消费/商家代理、CLI 子命令、账本格式。
7. **适配器等价**: 生成的 hooks 文件与 Node 版本输出一致。
8. **测试等价**: 使用 `pytest` 重新实现关键测试，确保行为一致。

### 1.4 关键约束

- **Python 3.12+**: 使用 `asyncio.TaskGroup`、类型参数语法 (`list[int]`)、`tomllib` 等特性。
- **uv 包管理**: `pyproject.toml` 使用 `[project]` + `[dependency-groups]`，可执行 `uv run evolver`。
- **跨平台**: Windows / macOS / Linux，正确处理路径、进程锁、信号差异。
- **核心文件混淆**: 原始 Node.js 版本中的 `src/evolve.js`、`src/evolve/pipeline/*.js`、`src/gep/a2aProtocol.js`、`src/gep/solidify.js` 等被 `javascript-obfuscator` 混淆。Python 移植基于测试契约 + 原始 Node.js 版本调用点 + 公开接口，重构实现而非逐行翻译。

### 2.6 进化流水线阶段

## 3. Python 版本技术选型

### 3.1 包管理

- **工具**: `uv` (>=0.5)
- **配置**: `pyproject.toml` 使用标准 `[project]` 元数据
- **依赖组**: `[dependency-groups]` 区分 `dev` / `test`
- **脚本**: `[project.scripts]` 注册 `evolver = "evolver.cli:main"`
- **入口**: `src/evolver/__main__.py` 支持 `python -m evolver`

### 3.2 核心依赖

| 用途 | 包 | 版本 |
|---|---|---|
| CLI 解析 | `click` 或 `argparse` | latest |
| 异步运行时 | `asyncio` (stdlib) | 3.12+ |
| HTTP 客户端/服务器 | `httpx` + `fastapi` / `aiohttp` | latest |
| 数据验证 | `pydantic` >=2 | latest |
| 设置管理 | `pydantic-settings` | latest |
| 环境变量 | `python-dotenv` | latest |
| 加密 | `cryptography` | latest |
| Git 操作 | `GitPython` (可选) + `subprocess` | latest |
| 文件锁 | `filelock` | latest |
| 进程信息 | `psutil` | latest |
| 测试 | `pytest`, `pytest-asyncio`, `pytest-cov` | latest |
| 代码质量 | `ruff`, `mypy` | latest |

### 3.3 为什么不选某些包

- **不选 `requests`**: `httpx` 同时支持 sync/async、连接池、SSE、HTTP/2，等价替换 Node `undici`。
- **不选 `flask`**: FastAPI 原生支持 async，与 Proxy 的心跳/sync engine 更契合。
- **不选 `trio`/`anyio`**: 保持简单，Python 3.12 `asyncio` 已足够。
- **不选 `sqlalchemy` / `sqlite`**: 原始项目使用 JSON/JSONL，保持文件语义等价。但 Python 版引入了 `ops/sqlite_store.py` 作为增强。

### 4.2 Node.js 版本独有的关键能力

以下能力在 Node.js 原版中已成熟，但 Python 移植版**完全缺失或仅为骨架**：

#### 4.2.1 工程韧性（多轮迭代回归测试）

Node.js 版对关键路径采用**多轮迭代回归**策略，而非一次性测试：
- `heartbeatResilienceRound{3,4,5,6,8,9}.test.js` — 心跳系统经过至少 7 轮迭代加固
- `forceUpdate*` 系列 8 个测试文件 — 自更新系统具有工业级的故障处理（并发守卫、幂等性、复制楔形检查 `midCopyWedge`、心跳触发更新、最后更新上报协议）
- `lifecycle*` 系列 10+ 文件 — 节点身份和生命周期管理有复杂的兼容层（遗留回退、统一、速率限制、过期密钥）

Python 版目前仅有 `test_ops_lifecycle.py`（13KB，较深入）和单个 `test_force_update.py`，远未复现上述韧性测试矩阵。

#### 4.2.2 AWS Bedrock 代理

Node.js 版 `proxy/router/messages_route.js` 包含 220+ 行的 `_proxyBedrock()` 实现，处理：
- 模型 ID 规范化（`canonicalizeForBedrock`）
- Claude Code adaptive thinking 降级处理（`adaptive` → `enabled`/`disabled`）
- AWS SDK v3 流式响应转标准 SSE
- 连接池复用与超时控制
- Token 复用优化（`proxyTokenReuse.test.js`）

Python 版虽有 `boto3` 依赖，但 proxy 模块结构更简单，没有同等级别的 Bedrock 流式处理代码。`/v1/messages` 路由在 handler 未配置时返回 `503`。

#### 4.2.3 WebUI 浏览器客户端

Node.js 版 `src/webui/client/` 包含完整的**浏览器端单页应用**：
- `clientJs/`：8 个模块（bootstrap、overview、pipelines、interactions、assets、personality、i18n、common）
- `indexHtml.js`、`stylesCss.js`、`static.js`：内联 HTML/CSS/静态资源
- `vendor/echarts.min.js`：内置图表库
- 支持实时 SSE 更新、多语言、响应式布局

Python 版仅有后端 Dashboard（`app.py` / `dashboard.py` 返回自包含暗色 HTML），没有任何对应的浏览器端交互逻辑。

#### 4.2.4 A2A 资产治理工作流

Node.js 版 `scripts/` 包含完整的 A2A 资产治理：
- `a2a_promote.js`：提升操作必须显式携带 `--validated` 标志；对 Gene 类型校验 `validation` 命令白名单；与本地基因 ID 冲突时保留本地版本并发射 `reject` 决策
- `a2a_ingest.js`：外部资产进入隔离区前通过 `contentHash.verifyAssetId()`，并向协议层发射 `quarantine` 决策
- `a2a_export.js`：支持 `--protocol` 模式逐条输出 A2A Publish 消息，支持 `--hello` 握手包

Python 版仅有基础的 `a2a_export.py`（40 行）和 `a2a_ingest.py`（51 行），无 promote/隔离区/quarantine 工作流。

#### 4.2.5 生产构建与分发

Node.js 版 `scripts/build_binaries.js` 具有军工级构建特性：
- 三阶段流水线：`bun bundle` → `javascript-obfuscator`（RC4 加密、控制流扁平化）→ `bun compile`
- 非确定性容错：混淆器偶发语法错误，脚本实现 **12 次自动重试**
- 跨平台矩阵：5 个目标平台，输出 SHA256SUMS，smoke test

Python 版无对应能力。需考虑 `PyInstaller`/`nuitka` 构建独立可执行文件。

#### 4.2.6 金丝雀与回滚安全

Node.js 版 `canary.js` 在 solidify 前 fork 子进程验证 `index.js` 可加载，防止语法错误导致守护进程崩溃。Python 版 `canary.py` 存在但完成度待验证。

Node.js 版有 `rollbackSafety.test.js` 专门测试 stash 模式下用户并行创建的文件不被误删（mtime 守卫）。Python 版缺少此类专项测试。

#### 4.2.7 人格分析与 ship gate

Node.js 版 `scripts/gep_personality_report.js` 实现：
- 双源聚合：合并 `model.stats` 与 `events.jsonl` 的 ground truth
- Laplace 平滑：`score = p(success) * 0.75 + avg_score * 0.25 * sampleWeight`
- 突变维度拆解：repair / optimize / innovate 三类意图的成功率独立统计

Node.js 版 `scripts/recall-verify-report.js` 实现质量门控：
- 任何 `asset_type` 的 `mismatch > 0` 直接判为 **RED**
- 成功率 < 0.95 且 ≥0.85 为 **YELLOW**；< 0.85 为 **RED**
- 计算 p50/p95/p99 的 `latency_ms` 与 `age_at_verify_ms`

Python 版均无对应脚本。

#### 4.2.8 经济机制细节

Node.js 版 ATP 实现完整的经济闭环：
- **Reuse Attribution（P4-a）**：`shadow` 模式下附加重用归因块
- **预算控制**：日预算 (`dailyCap`)、单笔预算 (`perOrderCap`) 双重限制
- **冷启动安全**：前 10 单必须人工确认
- **Consent 溯源**：首次运行生成 `~/.evomap/atp-consent`
- **自动买家默认开启但可 ack 关闭**

Python 版 `auto_buyer.py` 虽有账本路径和同意开关，但核心决策逻辑（能力缺口 → ATP 服务搜索 → 预算评估 → 下单）为 `pass`。

#### 4.2.9 安全加固深度

Node.js 版有专门的安全回归测试（GHSA-jxh8-jh77-xh6g）：
- 沙箱执行器白名单：仅允许 `node` 二进制，**明确禁止 `npm`/`npx`**
- Node 标志黑名单：禁止 `-e`, `--eval`, `-p`, `--print`, `-r`, `--require`, `--loader`, `--import`
- Shell 元字符过滤：`;`, `&&`, `|`, `` ` ``, `$()`, `>`, `<`, `&`

Python 版 `sandbox_executor.py` 存在但安全模型未完整实现，缺少针对 Python 特定注入向量（如 `python -c`, `eval()`, `exec()`, `__import__`）的专项回归测试。

#### 4.2.10 特性标志系统

Node.js 版 `featureFlags.js` 实现三层特性开关（env > 磁盘 > 代码默认），支持 Hub 动态推送。Python 版 `gep/feature_flags.py` 已实现 **env (`EVOLVER_FF_*`) → `evolver/.config/disk_flags.json` → legacy `~/.evomap/feature_flags.json` → `DEFAULT_FLAGS`**；`proxy/router/features.py` 委托同一来源做路由门控（`llm_messages`→`enable_llm_review` 等）。**差距**：Hub 动态推送尚未接线。

### 4.3 Python 版本相对领先之处

尽管存在大量落后，Python 版在以下方面具有相对优势：

1. **更清晰的 Pipeline 目录结构**：Node 版 `evolve.js` 被混淆为单行 88KB，Python 版明确拆分为 `pipeline/collect.py` 等 6 个阶段文件，可读性更强。

5. **明文代码**：Python 版完全可读，便于社区贡献和审计（Node 版 52 个核心文件经混淆保护）。

## 7. 关键等价性保证

### 7.1 数据文件互操作

Python 实现必须能读取 Node 版本生成的文件，Node 版本也必须能读取 Python 版本生成的文件。保证措施：

- JSON/JSONL 使用相同字段名和默认值。
- 时间戳统一使用 ISO 8601。
- ID 生成使用 UUIDv7（与 Node `crypto.randomUUID` 一致）。
- SHA-256 canonicalization 完全相同。

**当前状态**: Schema 和 asset_store 已确保格式一致，但**尚未执行正式的互操作验证测试**。

### 7.2 环境变量

所有环境变量名、默认值、优先级完全复刻。`config.py` 提供单一可信源。

**当前状态**: ✅ 已完整复刻。

### 7.3 CLI 行为

命令名、参数、退出码、stdout/stderr 格式完全复刻。测试中使用 golden output 对比关键命令。

**当前状态**: ⚠️ 参数解析完整，但部分命令底层行为不一致或缺失。

### 7.4 文件路径

默认路径解析逻辑完全复刻。通过设置 `EVOLVER_REPO_ROOT`, `MEMORY_DIR`, `EVOLUTION_DIR` 等环境变量，测试可以重定向到临时目录。

**当前状态**: ✅ 已实现。

### 7.5 异步模型

Node 的 `async/await` + `Promise.race` 对应 Python 的 `async/await` + `asyncio.wait_for` + `asyncio.TaskGroup`。硬超时、取消、并发控制需仔细对齐。

**当前状态**: ⚠️ 基础异步框架就绪，硬超时和进程替换待完善。

### 7.6 进程管理

- Windows: 跳过 detached spawn，依赖外部 supervisor。
- macOS/Linux: 支持自杀重生 (`spawnReplacementProcess`)，使用 `subprocess.Popen(detach=True)`。
- 单实例锁：使用 PID + mtime 租约机制。

**当前状态**: ⚠️ `ops/lifecycle.py` 跨平台进程管理较完整，但 watch 模式、时钟跳变检测、停滞自动重启等细节待完善。

### 7.7 锁与并发

- 资产存储使用 `filelock`（跨进程）。
- Python 内部使用 `asyncio.Lock`（协程级）。
- JSON 写入使用 `tmp + os.replace`（原子）。

**当前状态**: ✅ 已实现。Windows 上 `os.replace()` 不能替换已打开文件，守护循环中需额外处理。

## 9. 风险与注意事项（更新版）

### 9.1 混淆核心

原始核心算法文件被混淆，无法直接翻译。风险在于重构实现可能与原始行为存在细微差异。缓解：

- 严格遵循测试契约。
- 记录每个重构模块的假设和不确定点。
- 使用模糊测试/属性测试覆盖边界情况。

### 9.2 平台差异

- Windows 不支持 `SIGCONT`、`fork()`、`fcntl`。
- Windows `os.replace()` 与 Linux 行为略有不同（不能替换已打开文件）。
- 路径大小写不敏感问题。
- **新增**: `ops/lifecycle.py` 的进程管理在 Windows 下依赖 `wmic`/`ps`，可能需要 `pywin32` 增强。
- **新增**: `gep/idle_scheduler.py` macOS 空闲检测（ioreg/IOKit）尚未实现。

缓解：

- 抽象平台差异到 `evolver.platform` 模块。
- CI 覆盖 Windows + Linux + macOS。

### 9.3 uv/Python 3.12 兼容性

- 确保所有依赖支持 Python 3.12+。
- 避免使用 `asyncore`、`imp` 等已移除模块。

### 9.4 性能

- JSONL 大文件读取使用 seek/tail，避免全量加载。
- Proxy 的 mailbox 使用批量 poll，避免高频率磁盘 IO。
- 心跳和 sync 定时器使用 `asyncio`，避免线程开销。
- **新增风险**: `memory_graph.py` 数据量增长可能导致查询变慢，需设计分区策略（按月分文件）。

### 9.5 安全

- 所有 shell 命令禁用 `shell=True`。
- 验证命令白名单限制可执行文件。
- sanitize 模块扫描环境变量值反向外泄。
- TLS 强制默认开启，仅 `EVOMAP_HUB_ALLOW_INSECURE=1` 可绕过。
- **新增风险**: Proxy 大量路由为 stub，若在生产环境启用可能产生意外行为。
- **新增风险**: Validator 沙箱执行器安全模型未完整实现，需补充 Python 特定注入向量防护（`python -c`, `eval()`, `exec()`）。

### 9.6 许可证不一致

Node.js 原版使用 **GPL-3.0-or-later**，Python 移植版当前为 **Apache-2.0**。若目标是"完全行为等价"的官方移植，许可证应保持一致，或明确说明分发的法律依据。
